## 1. 需求说明

### 1.1 需求背景

在电商行业，客户服务质量直接影响用户留存与复购率。传统人工客服面临人力成本高、响应延迟大、服务质量波动等问题。本产品旨在构建一个以**数据收集与整理**为核心的智能客服Agent，通过大模型驱动的意图识别、情绪感知与多轮对话管理，在高效解决用户问题的同时，持续沉淀用户画像数据，为企业精细化运营赋能。

### 1.2 核心目标

1. **高效服务闭环**：覆盖售前导购、售中订单查询、售后客诉处理三大P0场景，实现从用户提问到问题解决的全链路自动化
2. **数据资产沉淀**：在对话过程中实时抽取用户画像数据，构建结构化用户档案，为企业运营决策提供数据支撑
3. **智能体验升级**：支持意图识别、情绪感知、意图切换、异常恢复等高级对话能力，打造接近真人的服务体验

#### 1.5.2 情绪判断规则

| 情绪等级 | 触发条件 | 系统行为 |
|----------|----------|----------|
| 中性(neutral) | 默认状态 | 正常服务 |
| 不满(dissatisfied) | 检测到抱怨/不满关键词或语气 | 插入安抚话术，继续服务 |
| 愤怒(angry) | 连续2轮不满或检测到强烈负面情绪 | 安抚 + 提供转接人工按钮 |

#### 1.5.3 异常兜底规则

| 异常类型 | 处理策略 |

| 意图与当前流程不匹配 | 确认用户是否要切换意图 |

| 规则 | 说明 |
|------|------|

```

     |
     +--[意图置信度 >= 0.5]---> [意图路由]
     |                                |
     |               +---> 售前导购流程(pre_sale_flow)
```

## 3. 系统流程图

### 3.1 系统架构总览（当前实现）

> **架构演进说明**：为优化首字延迟（从~40s降至~3s），系统已从单一LangGraph流水线演进为**并行双端点架构**。LangGraph Agent保留作为完整流程的兜底方案（`/api/chat/send`、`/api/chat/stream`），前端默认使用快速通道。

#### 3.1.1 并行双端点架构（核心优化）

```
用户发送消息
    |
    +---> POST /api/chat/quick/stream   (流式回复端点，~3秒首字)
    |       - 跳过意图识别，直接生成回复
    |       - 使用 DIRECT_RESPONSE_PROMPT
    |       - SSE流式输出，用户即时看到文字
    |       - 完成后累加Token到会话
    |
    +---> POST /api/chat/analyze        (分析端点，~7秒)
            - 意图识别 + 情绪判断 + 槽位抽取
            - 使用精简版 ANALYZE_PROMPT
            - 更新会话状态（意图/情绪/槽位/画像）
            - 累加Token到会话
            - 结果异步更新到前端数据看板

    两个请求并行执行，互不阻塞！
    用户感知延迟 = max(流式首字, 分析完成) ≈ 3秒
```

#### 3.1.2 快速意图检测

系统内置模式匹配机制，无需LLM调用即可快速判断常见意图：

| 意图 | 匹配关键词 |
|------|-----------|
| order_query | 订单、物流、单号、发货、快递、ORD- |
| after_sale | 退货、退款、售后、质量问题、换货、投诉 |
| pre_sale | 推荐、买、商品、价格、优惠、折扣 |
| chitchat | 你好、嗨、在吗、聊天、天气、时间 |

匹配失败时回退到 `general`，由分析端点进行精确LLM判断。

```
|                          前端 (React)                              |
```

### 3.3 数据流转说明

1. **用户输入** → 前端通过HTTP POST发送到 `/api/chat`，携带 `session_id`、`message`
2. **意图识别** → Agent调用Kimi API，传入用户query + 会话上下文，获取意图分类 + 情绪标签 + 槽位抽取结果
3. **路由分发** → 根据意图ID路由到对应Handler
4. **槽位检查** → Handler检查当前流程所需槽位是否齐全，不齐全则生成反问
5. **数据获取** → 从假数据服务中按条件检索匹配数据
6. **回复生成** → 调用Kimi API生成自然语言回复
7. **画像更新** → 将本轮抽取到的用户画像槽位数据更新到会话存储
8. **响应返回** → 将回复文本 + 结构化数据（意图、情绪、画像、Token消耗）返回前端

## 4. 槽位设计

### 4.0 槽位设计总则

#### 4.0.1 设计理念

槽位(Slot)是支撑业务流程推进和用户数据沉淀的核心数据结构。本系统采用**双轨槽位体系**：

- **业务流程槽位**：面向当前对话流程，用于收集完成本次服务所需的关键信息，具有临时性（会话级别）
- **用户画像槽位**：面向长期数据资产，用于构建用户结构化档案，具有持久性（跨会话级别）

#### 4.0.2 槽位填充策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 显式反问 | Agent主动向用户提问收集槽位 | 必填槽位缺失时 |
| 隐式抽取 | 从用户自然语言中由LLM自动提取 | 用户query中已包含槽位信息时 |
| 上下文推断 | 基于已有槽位和对话历史推理填充 | 可从其他槽位推导时 |
| 默认值兜底 | 使用预设默认值填充 | 非必填槽位且无法获取时 |

#### 4.0.3 槽位收集优先级规则

1. **必填槽位优先**：先收集业务流程闭环所必需的槽位
2. **依赖槽位后置**：存在依赖关系的槽位按依赖顺序收集
3. **高价值槽位优先**：对用户画像价值高的槽位优先抽取
4. **自然对话优先**：每轮最多反问1-2个槽位，避免"查户口"式体验

#### 4.0.4 槽位填充策略详解

系统采用"隐式抽取优先、显式反问兜底"的双轨填充策略：

##### 策略一：隐式抽取（Implicit Extraction）

从用户自然语言中自动识别并提取槽位值，无需用户感知。

| 抽取方式 | 说明 | 示例用户输入 | 抽取结果 | 置信度 |
|----------|------|-------------|----------|--------|
| 关键词匹配 | 基于预定义关键词库进行规则匹配 | "我想买个**手机**，预算**5000**左右" | product_category="手机", budget_range="5000" | 0.85 |
| 正则提取 | 通过正则表达式提取结构化信息 | "订单号**ORD-20260502-002**" | order_id="ORD-20260502-002" | 0.95 |
| 语义推断 | 通过LLM理解语义后推断槽位值 | "给老婆买个生日礼物" | usage_scenario="送礼", gender推断="女" | 0.70 |
| 上下文继承 | 从对话历史中继承已确认的槽位值 | 上文已确认品类为"手机"，用户说"那华为的呢" | brand_preference="华为" | 0.80 |

##### 策略二：显式反问（Explicit Ask）

当隐式抽取失败或置信度过低时，系统主动向用户提问收集槽位。

| 反问类型 | 适用场景 | 反问原则 | 示例 |
|----------|----------|----------|------|
| 开放式反问 | 槽位可选值范围较大 | 给用户充分自由度 | "您想了解哪类商品呢？" |
| 选项式反问 | 槽位为枚举类型 | 提供2-4个选项降低用户认知负担 | "您是想看手机、电脑还是耳机呢？" |
| 确认式反问 | 隐式抽取置信度中等(0.5-0.7) | 先猜测再确认 | "听您提到华为，是想了解华为手机吗？" |
| 引导式反问 | 用户表达模糊 | 给出具体引导方向 | "方便说一下预算范围吗？比如3000-5000或者5000以上" |

##### 反问时机控制

| 控制规则 | 说明 |
|----------|------|
| 单轮反问上限 | 每轮最多反问1-2个槽位，避免连续提问 |
| 反问间隔 | 两次反问之间至少间隔1轮正常对话（含业务数据展示） |
| 反问优先级 | 必填槽位 > 高价值槽位 > 一般槽位 |
| 反问跳过 | 用户情绪为dissatisfied/angry时，本轮跳过反问，优先安抚 |
| 反问放弃 | 同一槽位反问2次仍未获取到有效值，标记为"用户不愿提供"，不再追问 |

##### 填充策略决策流程

```
[用户输入]
     |
     v
[隐式抽取] -- 关键词匹配 + 正则提取 + 语义推断 + 上下文继承
     |
     +--[置信度 >= 0.8]---> [自动填充槽位] --> [进入下一步]
     +--[置信度 0.5-0.7]---> [确认式反问] --> [用户确认/修正] --> [填充槽位]
     +--[置信度 < 0.5]---> [检查反问条件]
                              |
                              +--[满足反问条件]---> [显式反问]
                              +--[不满足反问条件]---> [暂存待填充列表，等待下轮]
```

#### 4.0.5 槽位置信度管理

每个槽位值都附带置信度评分，用于判断是否需要反问确认以及是否可用于业务决策。

##### 置信度等级定义

| 置信度区间 | 等级 | 含义 | 处理方式 |
|-----------|------|------|----------|
| 0.9 - 1.0 | 极高 | 用户明确提供，几乎无歧义 | 直接使用，无需确认 |
| 0.7 - 0.89 | 高 | 从明确语境推断，歧义较小 | 直接使用，可在回复中自然确认 |
| 0.5 - 0.69 | 中 | 有一定推断成分，存在误判可能 | 使用前进行确认式反问 |
| 0.3 - 0.49 | 低 | 推断成分较大，仅作参考 | 不直接使用，作为反问的引导依据 |
| 0.0 - 0.29 | 极低 | 几乎无依据的猜测 | 丢弃，不用于任何决策 |

##### 置信度影响因素

| 影响因素 | 影响方向 | 影响幅度 | 说明 |
|----------|----------|----------|------|
| 用户明确声明 | 提升 | +0.3~0.4 | 用户直接说出槽位值（如"我预算5000"） |
| 正则精确匹配 | 提升 | +0.2~0.3 | 结构化数据被正则精确捕获（如订单号） |
| 单次语义推断 | 降低 | -0.1~0.2 | 仅通过LLM推断，无明确关键词支撑 |
| 多轮交叉验证 | 提升 | +0.1~0.2 | 多轮对话中不同信息相互印证 |
| 与历史画像一致 | 提升 | +0.05~0.1 | 新抽取值与已存储画像一致 |
| 与历史画像矛盾 | 降低 | -0.2~0.3 | 新抽取值与已存储画像矛盾 |
| 用户情绪异常 | 降低 | -0.1 | 用户在愤怒/不满状态下可能夸大或贬低 |

##### 置信度更新策略

```
[新槽位值到达]
     |
     v
[与已有值比较]
     |
     +--[无已有值]---> [直接存储，置信度=新值置信度]
     |
     +--[已有值存在]
           |
           +--[新旧值一致]---> [置信度 = max(旧置信度+0.05, 新置信度), 上限0.95]
           +--[新旧值不一致]
                 |
                 +--[新置信度 > 旧置信度+0.2]---> [替换为新值]
                 +--[新置信度 ≈ 旧置信度]-------> [保留旧值，标记为"待确认"]
                 +--[新置信度 < 旧置信度]-------> [保留旧值，忽略新值]
```

#### 4.0.6 槽位冲突解决规则

当同一槽位在不同轮次或不同来源产生冲突值时，按以下规则解决：

| 冲突场景 | 解决规则 | 示例 |
|----------|----------|------|
| 用户明确修正 | 以最新明确声明为准，置信度重置为0.9 | 用户先说"预算3000"，后说"其实预算是5000" → 更新为5000 |
| LLM推断与用户声明冲突 | 用户声明优先，LLM推断丢弃 | LLM推断gender="女"，用户说"我是男的" → 以用户声明为准 |
| 跨流程槽位冲突 | 保留各流程独立值，不做强制统一 | 售前说预算5000，售后投诉时提到"花了一万多" → 分别记录，标记冲突 |
| 历史画像与新抽取冲突 | 新抽取置信度>0.8时覆盖历史，否则保留历史 | 历史consumption_level="高"，本轮推断为"中"(置信度0.5) → 保留"高" |
| 多槽位交叉矛盾 | 标记为"数据异常"，不自动修复，等待用户澄清 | consumption_level="低"但购买商品均价>8000 → 标记异常 |

##### 冲突检测与处理流程

```
[检测到槽位值冲突]
     |
     v
[冲突类型判断]
     |
     +--[用户主动修正]---> [直接更新，置信度=0.9]
     |
     +--[LLM推断 vs 用户声明]---> [用户声明优先]
     |
     +--[跨流程冲突]---> [保留独立副本，标记关联]
     |
     +--[历史 vs 新抽取]
     |     |
     |     +--[新置信度 > 0.8]---> [覆盖历史]
     |     +--[新置信度 <= 0.8]---> [保留历史，新值记录到观察列表]
     |
     +--[交叉矛盾]---> [标记异常，下轮对话中自然引导澄清]
```

#### 4.2.5 画像数据生命周期

```
[会话开始]
     |
     v
[加载已有画像] -- 44个槽位全部加载，实时槽位初始为"未知"/空，预设槽位有默认值
     |
     v
[对话进行中] -- 每轮对话实时抽取17个实时槽位，前端看板实时更新
     |           预设槽位保持不变（不主动收集）
     v
[会话结束] -- 画像数据持久化（Demo中保留在内存）
     |
     v
[下次会话] -- 加载上次持久化的画像数据作为初始值
```

**数据更新策略**：
- **实时槽位（17个）**：每轮对话都可能更新，置信度提升时覆盖旧值
- **预设槽位（27个）**：仅在外部数据源更新时变更（如订单完成后更新avg_order_value），对话中不修改
- **列表型槽位**（preferred_categories、brand_affinity、hobbies、lifestyle_tags）：增量追加，不覆盖已有值
- **枚举型槽位**：仅在置信度高于当前值时覆盖

**数据过期策略**：
- 高频更新槽位（preferred_categories、brand_affinity）：每次对话都更新
- 中频更新槽位（consumption_level、price_sensitivity）：每3次对话评估一次
- 低频更新槽位（age_range、gender、region等）：仅在置信度提升时更新
- 预设槽位：不自动过期，依赖外部数据刷新

### 4.3 槽位回填与修正机制

#### 4.3.1 回填机制

当用户在后续轮次中补充了之前缺失的槽位信息时，系统自动回填并触发关联更新。

| 回填场景 | 触发条件 | 系统行为 | 示例 |
|----------|----------|----------|------|
| 延迟补充 | 之前跳过的槽位在后续对话中被提及 | 自动回填槽位 + 置信度标记 + 关联槽位重新计算 | 第1轮跳过预算，第3轮用户说"对了预算5000" → 回填budget_range |
| 跨轮推断 | 多轮信息累积后达到推断阈值 | 置信度达标后自动回填 | 第1轮提到"华为"，第3轮提到"Mate" → 回填brand_preference="华为" |
| 流程回溯 | 意图切换后回到原流程 | 检查原流程槽位完整性，缺失则回填 | 售前→订单查询→回到售前，检查是否有新槽位信息 |

##### 回填触发流程

```
[检测到新槽位信息]
     |
     v
[判断所属流程]
     |
     +--[当前流程槽位]---> [直接填充]
     +--[历史流程槽位]---> [回填 + 标记"延迟填充"]
     +--[其他流程槽位]---> [存储到对应流程 + 不打断当前流程]
     |
     v
[关联槽位重新计算] -- 回填可能触发其他槽位的交叉推断
     |
     v
[置信度更新] -- 回填槽位的置信度标记为"延迟填充"类型
```

#### 4.3.2 修正机制

用户可以在任意轮次修正之前提供的槽位信息。

| 修正方式 | 说明 | 示例 | 系统行为 |
|----------|------|------|----------|
| 显式修正 | 用户明确说之前的信息有误 | "不对，订单号是ORD-20260502-002" | 更新槽位值 + 置信度重置为0.9 + 重新检索 |
| 隐式修正 | 用户在新对话中提供了与之前矛盾的信息 | 之前说预算3000，现在问"8000的手机怎么样" | 检测冲突 → 确认式反问 → 用户确认后更新 |
| 否定修正 | 用户否定了系统的确认式反问 | 系统："您是想看华为手机吗？" 用户："不是，我看小米" | 丢弃原推断值 + 提取新值 |

##### 修正影响范围

当某个槽位被修正后，可能产生连锁影响：

| 被修正槽位 | 可能影响的关联槽位 | 处理方式 |
|-----------|-------------------|----------|
| product_category | budget_range, brand_preference, preferred_categories | 重新评估关联槽位有效性 |
| order_id | 所有订单相关槽位 | 清空订单相关槽位，重新检索 |
| budget_range | consumption_level, price_sensitivity | 重新计算消费能力等级 |
| issue_type | expectation, 处理方案 | 重新匹配处理方案 |

### 4.4 跨流程槽位共享规则

不同业务流程之间存在槽位共享关系，避免用户在不同流程中重复提供相同信息。

#### 4.4.1 槽位共享矩阵

| 槽位 | 售前导购 | 订单查询 | 售后客诉 | 共享策略 |
|------|:---:|:---:|:---:|------|
| order_id | - | 写入 | 读取 | 订单查询流程填充后，售后流程可直接读取 |
| phone | - | 写入 | 读取 | 同上，订单查询时获取的手机号可供售后使用 |
| product_category | 写入 | - | - | 售前填充的品类偏好可写入用户画像 |
| budget_range | 写入 | - | - | 售前填充的预算范围可写入用户画像 |
| brand_preference | 写入 | - | - | 售前填充的品牌偏好可写入用户画像 |
| issue_type | - | - | 写入 | 售后独立槽位，不跨流程共享 |
| issue_description | - | - | 写入 | 售后独立槽位，不跨流程共享 |

#### 4.4.2 共享规则

| 规则 | 说明 |
|------|------|
| 写入者权威 | 槽位的"写入"流程拥有该槽位值的最终解释权 |
| 读取免确认 | 从共享槽位读取的值置信度>=0.8时，无需再次向用户确认 |
| 读取需确认 | 从共享槽位读取的值置信度<0.8时，需自然确认（如"您之前查询的订单ORD-xxx，是要处理这个订单吗？"） |
| 过期失效 | 共享槽位在会话结束后失效，新会话需重新获取 |
| 冲突通知 | 读取时发现共享槽位值与当前流程预期不符，提示用户确认 |

#### 4.4.3 共享流程示意

```
[售前导购流程]                     [订单查询流程]                     [售后客诉流程]
     |                                   |                                   |
     v                                   v                                   v
 product_category="手机" ---------> preferred_categories              order_id="ORD-xxx" --共享--> order_id="ORD-xxx"
 budget_range="5000-8000" --------> consumption_level                phone="5678" --------共享--> phone="5678"
 brand_preference="华为" ---------> brand_affinity
```

### 9.2 防提示词注入

在每次LLM调用前，对用户输入进行以下校验：

### 9.3 违规内容拦截

| 检测维度 | 规则 | 处理方式 |
|----------|------|----------|
| 平台引导词 | 检测"去淘宝""去京东""去天猫""去官网""自己去买""自己下单"等 | 自动替换为"好的，我这就帮您操作" |
| 操作推诿词 | 检测"自己操作""自己去""自己弄""自己搞"等 | 自动替换为"好的，我这就帮您操作" |
| 检测位置 | graph.py所有Handler、app.py快速响应模式 | 统一调用filter_forbidden_content函数 |

**后端实现代码（app.py + graph.py）：**

```python
# 违规关键词列表
FORBIDDEN_KEYWORDS = [
    "去淘宝", "去京东", "去天猫", "去其他平台", "去官网", 
    "自己去买", "自己下单", "自己操作", "自己去", "自己弄", "自己搞",
    "到淘宝", "到京东", "到天猫", "到其他平台", "到官网",
    "去买", "自己买", "自己付", "自己支付", "自己付款"
]

def filter_forbidden_content(reply: str) -> str:
    """拦截并替换违规内容"""
    for keyword in FORBIDDEN_KEYWORDS:
        if keyword in reply:
            return "好的，我这就帮您操作😊"
    return reply
```
