## 2. 项目架构

### 2.1 整体架构

```
+-------------------+          HTTP/SSE          +-------------------+
|     前端 (React)   | <----------------------> |   后端 (FastAPI)   |
|  Create React App  |                          |     Uvicorn        |
+-------------------+                          +-------------------+
                                                        |
                                                        v
                                               +-------------------+
                                               | LangGraph Agent   |
                                               | 状态图编排         |
                                               +-------------------+
                                                        |
                                                        v
                                               +-------------------+
                                               |   Kimi API (LLM)  |
                                               | moonshot-v1-8k    |
                                               +-------------------+
```

### 2.2 并行双端点架构

系统采用并行双端点设计，将回复生成与分析解耦，首字延迟从约 40 秒优化至约 3 秒：

```
用户发送消息
    |
    +---> POST /api/chat/quick/stream   (流式回复，~3秒首字)
    |       - 跳过意图识别，直接生成回复
    |       - SSE 流式输出
    |
    +---> POST /api/chat/analyze        (分析端点，~7秒)
            - 意图识别 + 情绪判断 + 槽位抽取
            - 更新会话状态与用户画像
            - 结果异步更新到前端数据看板

    两个请求并行执行，互不阻塞
```

## 3. 核心模块说明

### 3.1 LangGraph Agent 编排

使用 LangGraph StateGraph 构建多节点状态机，支持以下节点：

| 节点 | 功能 |
|------|------|
| intent_classifier | 意图识别 + 情绪判断 + 槽位抽取 |
| emotion_analyzer | 情绪处理决策 |
| router | 意图路由分发 |
| pre_sale_handler | 售前导购处理 |
| order_query_handler | 订单查询处理 |
| after_sale_handler | 售后客诉处理 |
| general_handler | 通用咨询处理 |
| chitchat_handler | 闲聊处理 |
| fallback_handler | 异常兜底处理 |
| response_generator | 回复生成 |
| profile_updater | 用户画像更新 |

### 3.2 记忆系统

| 类型 | 存储方式 | 生命周期 | 内容 |
|------|----------|----------|------|
| 短期记忆 | 内存 Dict | 会话级别 | 对话历史、当前意图、情绪、流程状态 |
| 用户画像 | 内存 Dict | 会话级别 | 40 个画像槽位（基础画像、消费特征、服务交互、兴趣生活、行为特征） |
| 流程快照 | 栈结构 | 会话级别 | 中断流程的状态快照，支持恢复 |

### 3.3 API 端点

| 端点 | 方法 | 功能 |
|------|------|------|
| /api/session/create | POST | 创建新会话 |
| /api/session/{id} | GET | 获取会话信息 |
| /api/session/{id} | DELETE | 删除会话 |
| /api/chat/send | POST | 发送消息，获取完整回复（LangGraph 完整流程） |
| /api/chat/stream | POST | 发送消息，获取流式回复（LangGraph 完整流程） |
| /api/chat/quick | POST | 快速响应模式（跳过意图识别） |
| /api/chat/quick/stream | POST | 快速响应模式（流式） |
| /api/chat/analyze | POST | 独立分析端点（意图+情绪+槽位+Token） |
| /api/panel/{id} | GET | 获取数据看板信息 |

### 3.4 安全机制

| 机制 | 实现方式 |
|------|----------|
| 限流 | 单会话每分钟最多 10 次调用，IP 级别限流 |
| 防提示词注入 | 输入校验，检测恶意注入模式 |
| 敏感信息脱敏 | 手机号等敏感信息脱敏展示 |
| API Key 管理 | 环境变量管理，不硬编码 |

## 4. 数据流

### 4.1 用户消息处理流程

```
用户输入 → 前端并行发起两个请求
    |
    +---> quick/stream → 直接生成回复 → SSE 流式返回 → 前端逐字显示
    |
    +---> analyze → LLM 意图识别 → 情绪判断 → 槽位抽取 → 更新会话状态
                                              |
                                              v
                                        更新前端数据看板
```

### 4.2 用户画像抽取流程

```
用户输入 → analyze 端点 → LLM 分析 → profile_slots
    |
    +---> translate_profile_slots() → 英文值转中文
    |
    +---> session_store.update_user_profile() → 更新会话画像
    |
    +---> 前端 /api/panel → 获取最新画像 → DataPanel 展示
```

## 7. 关键技术选型说明

### 7.1 为什么选择 FastAPI 而非 Flask？

- 原生异步支持，与 LangGraph 的异步调用天然契合
- 自动 API 文档生成（Swagger UI）
- 基于 Pydantic 的数据校验，减少手动校验代码
- 性能优于 Flask，适合高并发场景

### 7.2 为什么选择 LangGraph？

- 状态图模式清晰表达多轮对话的状态流转
- 支持条件边和动态路由，适合意图切换场景
- 与 LangChain 生态无缝集成
- 可视化调试能力

### 7.3 为什么选择 Kimi API？

- 国产大模型，中文理解能力强
- OpenAI 兼容接口，接入成本低
- 性价比高，适合客服场景

### 7.4 为什么采用并行双端点架构？

- 传统 LangGraph 串行流程首字延迟约 40 秒
- 并行架构将回复生成与分析解耦
- 用户感知延迟降至约 3 秒，体验大幅提升
